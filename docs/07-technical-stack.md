## Stack choices

| Layer | Choice | Why | Alternative considered |
|---|---|---|---|
| Backend language/runtime | Python 3.12 | RAG and multi-agent LLM orchestration are now core, not a delegated side call — Python has the deepest ecosystem for both (embedding SDKs, agent frameworks, vector-DB clients), and it's a natural fit for an I/O-bound, LLM-call-heavy backend. | TypeScript/Node (prior choice) — rejected now that the system's central logic is RAG retrieval and multi-agent orchestration rather than CRUD-plus-webhook logic; Python's ecosystem maturity for that specific workload outweighs the prior "one language across stack" argument. |
| Backend framework | FastAPI | Async-native (needed for concurrent LLM/embedding I/O), Pydantic models give typed structured validation that maps directly onto structured LLM outputs and RAG payloads, automatic OpenAPI schema for the separate frontend to consume. | Django — batteries-included but a sync-first ORM and heavier framework overhead not justified for an API-first backend whose primary job is orchestration, not server-rendered pages. |
| Frontend framework | Next.js (React) — unchanged | Frontend and backend are now fully separate deployables in different languages, which directly satisfies the "separate backend" requirement; Next.js remains the right fit for a data-dense HR dashboard and stays a pure API client of the FastAPI backend. | A Python server-rendered framework — rejected, would recouple frontend to backend language and lose React's component/table ecosystem suited to this UI. |
| Primary database | PostgreSQL + `pgvector` extension | Keeps structured data and the vector index in one system so the existing RLS-based multi-tenancy design (I2, extended to I11 in [04-invariants.md](04-invariants.md)) covers embeddings without a second isolation mechanism to build and verify. | Dedicated vector database (Qdrant / Weaviate / Pinecone) — better large-scale ANN performance and richer hybrid search, rejected for v1 specifically because it introduces a second stateful service with its own tenant-isolation surface; full reasoning in [06-architecture.md](06-architecture.md). |
| Embeddings model | Voyage AI (`voyage-3`, 1024-dim) | Purpose-built retrieval embeddings with strong domain recall; Anthropic's recommended embeddings partner, keeping the AI vendor surface small (two providers total, both used elsewhere in this stack). | Self-hosted open-source embedding models (e.g., `sentence-transformers`) — rejected for v1: avoids standing up and maintaining embedding inference infrastructure, consistent with the "no self-hosted ML infra" principle below. |
| LLM crew / agent orchestration | CrewAI | Purpose-built for exactly this shape of problem — role-based agents ("Extraction Agent," "Summarizer Agent," "Reasoning Agent") collaborating as a defined crew, with each agent bindable to a different underlying model by task complexity. Directly matches the "multi-model LLM crew" requirement this stack is built around. | LangGraph — more flexible low-level graph control over agent state and transitions, rejected for v1 because it requires materially more custom orchestration code to reproduce the same role-based crew pattern CrewAI provides out of the box; revisit if CrewAI's higher-level abstraction becomes limiting for more complex multi-step reasoning. |
| LLM models (multi-model assignment) | Claude Haiku 4.5 — Extraction Agent (resume field extraction, high volume/low complexity per call); Claude Sonnet 5 — Summarizer Agent (resume + scorecard synthesis, balanced cost/quality); Claude Opus 4.8 — Reasoning Agent (candidate-to-requisition match rationale, lowest volume/highest stakes) | Matches model capability and cost to task complexity and call volume — the extraction step runs on every resume (needs to be cheap and fast), while the reasoning step runs only on retrieved candidates for an active search (low volume, worth paying for the strongest available reasoning). | A single model for every crew role — rejected: wastes spend running high-volume extraction on the most expensive model, or under-powers the highest-stakes reasoning step if the cheapest model is used everywhere. |
| Async task queue | Celery + Redis | Mature multi-step task chaining/canvas primitives needed for the parse → embed → (on-demand) crew pipeline described in [06-architecture.md](06-architecture.md); Python-native, well-understood retry/backoff semantics for LLM-call failures. | RQ — simpler API, rejected because it lacks Celery's chaining/canvas features needed to cleanly express the multi-step ingestion and crew pipelines; Dramatiq — comparable capability, rejected mainly for Celery's larger ecosystem and hiring-familiarity advantage at this stage. |
| Object storage | S3-compatible storage (AWS S3) — unchanged | Standard for file storage with signed-URL access patterns feeding the namespaced-key isolation approach in [06-architecture.md](06-architecture.md). | Storing files as DB blobs — rejected outright, as before. |
| Authentication (HR users) | Managed auth provider (Auth0/Clerk) with org-scoped sessions — unchanged in approach | JWT validation from FastAPI is no harder than from any other framework; org_id embedded in the session maps directly to the RLS session variable. | Custom-built auth — still rejected for the same security-surface reasons as the original stack decision. |
| Authentication (candidates) | Magic-link / time-limited email token, no password — unchanged | Matches A9 in [02-assumptions.md](02-assumptions.md). | Full candidate account system — still out per the Scope Creep Watchlist. |
| Hosting | AWS: ECS Fargate for the FastAPI API and all Celery workers (parsing, embedding, crew, notification run as separate task definitions for independent scaling), RDS for PostgreSQL/pgvector, S3 for objects, ElastiCache for Redis | Same managed/serverless-adjacent posture as the original stack decision, retargeted at a Python runtime; separate Fargate task definitions per worker type let the crew workers (LLM-call-bound, need higher timeout/retry tolerance) scale independently from the parsing/embedding workers. | Self-managed Kubernetes — still rejected for v1: operational overhead not justified at target scale, and now doubly true given the team also has to own the RAG/crew pipeline's own operational complexity. |
| Email delivery | Transactional email API (Postmark/SES) — unchanged | Deliverability/bounce handling is a solved problem. | Self-hosted SMTP — still rejected. |

## Choices made specifically to keep v1 scope tight

- **No microservices split, despite the added AI surface.** The API and all Celery workers (parsing, embedding, crew, notification) are separate deployable processes for independent scaling, but remain one Python codebase, one repo, one data-access layer. It would be tempting to carve the LLM crew into its own service given how different its dependencies are (CrewAI, model SDKs) from the CRUD API — resisted for v1 for the same reason as the original stack doc: it multiplies the surface area I2/I11 must be re-verified across, for a team-scaling problem this project doesn't have yet.
- **No dedicated vector database.** Restated from [06-architecture.md](06-architecture.md): pgvector inside the existing PostgreSQL instance, not a second stateful system, is the v1 default specifically to avoid a second tenant-isolation surface.
- **No self-hosted ML/NLP or embedding infrastructure.** Both the LLM crew and the embedding step call hosted provider APIs (Anthropic, Voyage) rather than standing up model-serving or GPU infrastructure — avoids owning inference infrastructure that is undifferentiated relative to the actual product problem.
- **No custom auth.** Unchanged rationale from the original stack decision.
- **A fixed three-model crew assignment, not a configurable/pluggable model registry.** Making the Extraction/Summarizer/Reasoning model bindings configurable per organization or swappable at runtime would be premature flexibility before any organization has asked for it — the fixed mapping above is the whole v1 answer.

## Open Questions

- Does the Claude API (used for extraction, summarization, and reasoning) need a fallback/secondary provider for availability, especially now that a live user-facing search flow (not just background parsing) depends on it synchronously-from-the-user's-perspective?
- Should CrewAI's agent definitions live in the same FastAPI/Celery codebase, or is there a case for packaging the crew as an independently versioned/deployed component given it's likely to iterate faster (prompt/model tuning) than the core CRUD API?
- At what resume/chunk volume does pgvector's ANN index need tuning (HNSW parameters) or does the team need to seriously plan the migration path to a dedicated vector database discussed and deferred in [06-architecture.md](06-architecture.md)?
- Is Voyage AI's embedding dimension (1024) and model version something to pin with a documented migration plan, given changing it later requires re-embedding every `resume_chunks` row?
